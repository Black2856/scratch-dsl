# 実装から得た教訓（soul-beat / pixel_survivors）

このリポジトリの DSL で実際に 2 本の大きな作品を作って判明した、**ドキュメントには書いていない
挙動・設計判断・作業手順の落とし穴**をまとめる。

- `workspace/soul-beat/` — DELTARUNE 風の音ゲー（66曲/404譜面、選曲・設定・STATUS・セーブ画面つき）
- `workspace/pixel_survivors/` — Vampire Survivors ライクの見下ろしアクション（ペン stamp エンジン）

役割の違い:

| ファイル | 内容 |
|---|---|
| `AGENTS.md` | リポジトリ全体の規約・作業フロー |
| `VIBE_CODING_REFERENCE.md` | DSL の書き方（形式・ビルダー・opcode・id 規則） |
| **この文書** | 実際に動かして分かった**実行時の挙動と設計の勝ち筋/負け筋** |

各項目の末尾に、判明した時期と対象プロジェクトを添えた。

## 1. Scratch VM / renderer の実行時挙動

### 1.1 warp proc 内の `broadcast and wait` は busy-spin する

warp（画面を再描画せずに実行する）プロシージャの中で `event_broadcastandwait` を呼ぶと、
sequencer が warp スレッドの yield を同ステップ内で即再実行するため受信側が走れず、
**WARP_TIME（500ms）いっぱい空回りする**。1 依頼ごとに VM 全体が約 0.5 秒停止し、
soul-beat では 60fps → 2fps まで落ちて判定が数秒遅延した。

- `broadcast and wait` は**非 warp スクリプト専用**（起動シーケンス等）にする。
- プレイ中の動的クローン生成は次のどちらかにする:
  - **プル方式** — テンプレート自身がローカルカーソルで共有リストを走査して自律クローンする
  - **メールボックス方式** — レーン別のグローバルセル（`fxKind`/`fxParent` 等）を受信側が毎フレーム消費する

*(2026-07-02 / soul-beat)*

### 1.2 クローン上限は実 VM で 300 固定・超過は無音で失敗

エラーも警告も出ず、ただクローンが生まれない。表形式のテキスト（STATUS の RECORDS 表は
最大 12 行 × 約 50 文字 = 450 個）はクローンでは組めない。→ §3.1 の stamp 方式へ。

*(2026-07-07 / soul-beat)*

### 1.3 文字列比較は大文字小文字を区別しない

`Cast.compare` が両辺を `toLowerCase()` してから比較する（標準 Scratch v14.1.0 の仕様であって
DSL 固有ではない）。影響を受けるのは `=` `<` `>` `item # of (list)` `list contains`。

- 置換表を `itemNum(alphabet, ch)` で引く設計で、アルファベットに `a` と `A` を両方入れると
  位置が曖昧になり**可逆でなくなる**。セーブコードの codec は小文字のみ 42 字にした。
- 大文字が必要なフィールドは codec を通さず、**復元後に別の一意変換で導出**する。
- 一般に「大小が畳まれる」前提で設計する。厳密に区別したいなら文字コード変換が要るが、
  Scratch に該当ブロックは無い。

*(2026-07-06 / soul-beat)*

### 1.4 `switchCostume` に数値型を渡さない

Scratch は数値を**コスチューム番号（インデックス）**として解釈する。桁表示などで数値を
そのまま渡すと別の絵に化ける。必ず `op.join('', op.mod(...))` で文字列化する。
（サブエージェントに委譲した際、この `join` を「冗長」と判断して外され桁ズレしかけた実績あり）

*(2026-07-16 / soul-beat)*

### 1.5 フェンシング（画面端クランプ）は理論計算でなく実測で確かめる

`getFencedPositionOfDrawable`（`FENCE_WIDTH=15`、`inset=floor(min(w,h)/2)`）の挙動を事前計算し
「タイル 10 枚目はクランプされる」と判断して 9 枚に妥協したが、実際に座標を取ったら
クランプは起きず 10 枚とも要求どおり配置できた。

原因は `keepInFence` が `dy = newY - drawable._position[1]`（**直前位置からの相対移動量**）を
使うこと。連続する `gotoxy` が絡むと、絶対座標だけの「中心 ± 半径 vs ステージ境界」計算では
実挙動を予測できない。

→ 画面端近くにクローンを並べるときは、諦める前に
`node --no-warnings --experimental-strip-types tools/turbowarpShot.ts <name> --wait N` で
実クローンの `state.sprites[].y` を取得して検証する。

*(2026-07-02 / soul-beat)*

### 1.6 pen は最背面・480×360 固定解像度

レイヤーグループが `background → video → pen → sprite` 固定なので、pen はどのスプライト
クローンよりも必ず奥になる。ただし**枠 PNG の内部が透明なら背面の pen 線は見える**ので、
「背面で構わない描画」には使える（選曲画面のスクロールバー・ジャンル見出しの飾り線で採用）。

さらに重要な制約として、**スプライトは表示解像度で再描画されるが pen バッファは常に 1x
（480×360）に焼かれる**。

- 小さい文字の stamp は実機で判読不能（サイズ 22〜30 ≈ 実質 7〜10px で潰れた）。
  **stamp テキストはサイズ 37 以上**（≈12px+）に限り、細かい表や小さいラベルはクローンで描く。
- stamp する文字は**整数ピクセル整列が必須**。非整数スケール（18×34 グリフ × 44% = 7.92×14.96px）
  だと最上段がサンプリングで落ちて上端が欠ける。①グリフが整数 px 寸法になるサイズを選ぶ、
  ②寸法が奇数 px なら中心座標を `*.5` に置いて境界に合わせる。25px 以上なら誤差が分散するので
  気にしなくてよい。
- 画面を抜けるときは必ず `penClear`（消し忘れると次の画面に残る）。
- pen は hidden スプライトからでも描ける。ただし **`pen_stamp` は show 中でないと写らない**。

*(2026-07-05 〜 07-07 / soul-beat)*

### 1.7 ターボモードの有無で見え方が変わる → warp を徹底する

非ターボでは**非 warp のループが 1 反復ごとに画面を再描画する**ため、クローンをループ生成すると
「1 体ずつ順に出現」して見える（ターボでは一括表示）。

- ビルダーの `defProc` は既定 warp=true。ただし **hat スクリプト直下のインラインループ
  （`op.repeat`/`op.until`）は warp にならない**。クローン生成ループは必ず warp proc に包む。
- cross-sprite 生成は broadcast が要るが、受信ハンドラ内の生成ループも warp proc 化して
  1 フレームで完了させる（ただし warp 内の `broadcast and wait` は §1.1 のとおり厳禁）。

*(2026-07-05 / soul-beat)*

## 2. フレームレート非依存で書く

生成した `.sb3` は **TurboWarp や拡張機能で開けば 30fps を超えて動く**。標準 VM の 30fps に
縛られた設計にしない。

- 時間に関わる量（判定の時刻差・ノーツ落下位置・カウントダウン）はすべて `sensing_timer`
  （ms 精度）基準で計算し、**フレーム数で数えない**。高 fps ほど入力検知が細かくなり、
  設計を変えずに精度と滑らかさが上がる。
- **毎フレーム定率で寄せる指数イージング（`x += (target - x) * k`）は禁止**。60fps では 30fps の
  倍速で動く。dt 補正した `factor = 1 - retain^(dt/16.667)`
  （= `1 - mathop('e ^', ln(retain)/16.667 * dt)`、dt は `timer*1000` の差分）を使うか、
  t0 基準の固定時間トゥイーンにする。soul-beat の SOUL カーソルがこの罠にかかった。
- 判定窓のような「fps に依存して体感が変わる定数」は `config/constants.ts` に出して可変にする。

*(2026-06-24 / 2026-07-05 / soul-beat)*

## 3. アーキテクチャの選択

### 3.1 大量エンティティは「クローン」より「並列リスト + pen stamp」

pixel_survivors は v1 をクローン方式で作ったあと、**v2 でクローンを全廃**して作り直した。

- エンティティは Stage の並列リスト（`e*`/`p*`/`g*`/`d*`/`f*` 系）で保持する。
- engine スプライト 1 体の warp proc `frame` が「更新 → erase → 全 stamp/pen dot 描画」を
  毎フレーム原子実行する。
- stamp は数値のコスチューム index（CI マップ）で切り替える。
- 時限系は毎フレーム減算せず**タイムスタンプ比較**にする。衝突判定は全て距離の二乗で行う。
- スプライトとして残すのはクリック UI とバナー・効果音用だけ。

結果、クローン上限（§1.2）とレイヤ順の問題から解放され、敵 64 体 + 弾 128 + ジェム 1023 +
ドロップ 255 を 1 スプライトで捌けるようになった。地面スクロールは 960×720 タイルを 1 stamp、
代表位置を `[-240,240)` に正規化してフェンシングを回避している。

同じ発想は soul-beat の STATUS 画面でも使った。glyph スプライトの**オリジナル**が warp proc 内で
`show → (gotoXY/switchCostume/setEffect → pen_stamp) × N → hide` すれば、クローン 0 で 450 文字を
描ける（warp 中はフレーム描画が走らないのでスプライト自体は一瞬も映らない）。
再描画はスクロール等の変化時だけ `penClear` → 全再 stamp（静止中コストゼロ）。
ペンはクリッピングできないので、窓外の行は行ごとスキップしないと枠外に漏れる。

*(2026-07-07 / soul-beat, 2026-07-20 / pixel_survivors)*

### 3.2 スプライトのローカル変数は proc 間で共有される

engine のような「1 スプライトに多数の proc を合成する」構成では、ローカル変数は
**proc をまたいで同じ実体**である。proc 呼び出しを挟むフラグに汎用の一時変数（`t5`/`t6` 等）を
使うと、呼び先が同じ変数を潰す。pixel_survivors では `hitE` が呼び出し元の一時変数を壊す事故を
2 回起こした。→ **呼び出しをまたぐ値は専用変数**（`flagA`/`flagB`/`n`/`kb`）にする。

*(2026-07-20 / pixel_survivors)*

### 3.3 `defProc` は `callProc` より前に置く

ビルダーの制約。ファイル内の定義順に注意する（分割したモジュールの import 順にも効く）。

*(2026-07-20 / pixel_survivors)*

## 4. データとアセットの設計

### 4.1 永続データのキーは「内容から導出した ID」にする

soul-beat のスコア記録は、曲や難易度の**並び順ではなく譜面ヘッダ由来の 16hex `chartId`**を
キーにしている。この設計のおかげで、曲を途中に挿入して全曲の表示番号がずれても、
**ユーザーのセーブ記録は一切壊れない**（実際に 65 曲目を 32 番へ挿入して確認済み）。

逆に、**配列順に依存する値（音源 index `bgm`、`chartSel` の通し番号）は挿入で全部ずれる**。
これらは手で書かず、`charts/index.ts` が配列順から自動付与し、`scripts/gen-song-music.mjs` が
音源を再採番する構造にしてある。曲を足したら**必ず gen スクリプトを流し直す**。

*(2026-07-23 / soul-beat)*

### 4.2 アセットは生成スクリプトで冪等に作る

pixel_survivors のアセット 343 エントリは `scripts/gen-assets.mjs` が全生成する
（原作の `sprites.js` を `node:vm` で実行してピクセル完全一致、5×7 の自作フォント、
`audio.js` のシンセを WAV 化 20 種）。依存なし・冪等なので、いつ流しても同じ結果になる。
手で並べた画像をリポジトリに置くより、**生成器を置くほうが差分レビューも再現も楽**。

*(2026-07-20 / pixel_survivors)*

### 4.3 表示できる文字はスプライトごとのグリフ集合に縛られる — しかも黙って化ける

テキストはグリフのコスチュームを `switchCostume` で切り替えて描くので、
**そのスプライトの `glyphCostumes(prefix, chars)` に無い文字は表示できない**。
そしてコスチュームが見つからないとき Scratch は**エラーを出さず直前のコスチュームのまま**になる。

実例: STATUS 画面の文字集合に `!` が入っておらず、既存曲の `DATING START!` `DUMMY!` が
静かに壊れていた（曲追加で気づいた）。

- 素材が無い文字（`'` `,` など）は曲名・ラベルから**あらかじめ落とす**（原題はコメントに残す）。
- **表示データと文字集合の突き合わせを検証スクリプトで自動化する**。TypeScript 側でメタデータを
  持っているので、ビルド前に「全表示文字 ⊆ 各スプライトの文字集合」を検査できる。

*(2026-07-23 / soul-beat)*

### 4.4 グリフは彩色素材 1 本 + brightness で表現する

白グリフ（`g_*`）と彩色グリフ（`gc_*`）を二重に持つのをやめ、彩色素材だけを保持して
brightness エフェクトで白を作る方式に統一した（コスチューム 653 → 447、ロード約 16% 短縮）。
正本は `workspace/soul-beat/src/lib/glyphs.ts`（等価変換の数学的根拠つき）。

- 白 = `setEffect('brightness', 100)`
- 淡色 = `color(N)` + `brightness 82`（color エフェクトの彩度 0.09 強制 quirk と厳密一致）
- 彩色 = `color(N)` + `brightness 0` を**明示**

**クローンは親のエフェクト状態を継承する**ので、暗黙の 0 に頼ると画像コスチュームが白化する。
スポーン前と毎フレーム更新の両方で必ず明示する。

*(2026-07-16 / soul-beat)*

### 4.5 音源を積むと `.sb3` が爆発する

soul-beat は 66 曲の mp3 で **`.sb3` が 221MB** になった（65曲時点で 96MB）。
`scratch-parser` は通るし TurboWarp でも動くが、**Scratch 本体へのアップロードは現実的に不可能**で、
ロード時間も相当に延びる。音数の多い作品では早い段階で「どこまで積むか / 再エンコードするか /
プロジェクトを分割するか」を決めておく。

*(2026-07-23 / soul-beat)*

## 5. ツールチェーンと検証

### 5.1 `npm run shot` はフラグを食う

`npm run shot -- <name> --wait 9000` は npm がフラグを飲み込んで既定 2500ms のままになる。
**node で直接呼ぶ**:

```
node --no-warnings --experimental-strip-types tools/turbowarpShot.ts <name> --wait 9000
```

### 5.2 preview サーバはプロジェクトをモジュールキャッシュする

`tools/turbowarpShot.ts` は既存の preview サーバ（port 8788）があればそれを再利用する。
preview サーバはプロジェクト TS をキャッシュするので、**`npm run sb3` で再ビルドしたら
turbowarpPreview のプロセスを kill して再起動**しないと古いプロジェクトが配信される。

### 5.3 テスト前に `.sb3` を再ビルドしたか必ず確認する

古いビルドに対してテストして「修正が効かない」「再現しない」と誤診断した事例が複数ある。
判定異常を疑う前にビルド時刻を見る。

### 5.4 キー入力は「ホールド」で送る

Playwright の `press()` は keydown+keyup を即発火するため、VM の毎フレーム `keyPressed`
ポーリングに拾われず**カーソル移動もスクロールも一切動かない**。移動系は
`down → 130ms 超 hold → up` で送る。broadcast ハンドラの起動だけなら tap でも足りる。

- `--keys` のキー名は **Playwright 表記**。`space` は `Unknown key` エラー（正しくは `Space`
  `Enter` `ArrowRight`、英数字は `1` `a`）。
- 連続キーの間隔は固定 150ms。**キー処理側に数フレームかかる初期化（非 warp の repeat 等）が
  あると後続キーを取りこぼす**（pixel_survivors で `runStart` の repeat(44) が 1.5 秒占有し、
  武器選択の数字キーが空振り → warp proc 化で解決）。遅延が要るならハンドラの無いダミーキー
  `x` を 150ms の間として挟める。
- `shot` の state 出力は **Stage のグローバル変数のみ**（スプライトローカルは出ない）。
  デバッグ計測値は一時的にグローバルへ置く。
- ヘッドレスでは `sensing_username` は空文字。

### 5.5 1x のスクリーンショットで字形を判断しない

「小さいグリフは非整数スケールで字形が壊れる（E→F、D→O）」と結論したが、**真因は
preview の canvas が 480×360 の固定物理解像度だったこと**だった。preview を
「表示サイズ × devicePixelRatio」の実解像度描画に直したら、**37.5% でも字形は完全に無傷**だった。

→ 小さい文字を評価するときは必ず `--zoom 2` で撮る。1x のスクショで「壊れた」と判断しない。
ただし 1x 環境（480×360 等倍表示）では実際に劣化する点は留意する。

*(2026-07-05 の訂正 / soul-beat)*

### 5.6 preview で再現しない挙動が実機で出る

本リポジトリの preview/shot は約 60fps で動く。**fps やフレームタイミングに依存するバグは
preview を徹底的に叩いても再現しないことがある**（soul-beat の設定画面「最下部で下キー → 10行
飛ぶ」は TurboWarp の高 fps 環境でのみ発生した）。再現しないときは環境差を疑う。

## 6. DSL の限界とその回避

### 6.1 クラウド変数は生成できない → フックだけ用意して手動で差し込む

DSL は `isCloud:false` 固定でクラウド変数を作れない。そこで
**「呼び出しの配線」だけ DSL で作り、クラウド変数への set ブロックだけを `.sb3` 生成後に
エディタで手動追加する**方式にした。

- フック proc（`recordCloudScore`）内で payload を組み立てて変数（`cloudPayload`）に入れるところまでを DSL で完結させる。
- `procedures_definition` に `DslComment` でエディタから見える目印を置く
  （「⚠ CLOUD VAR HOOK（DSL 未対応・手動追加）」）。
- 手動作業を**1 ブロックだけ**に押し込めるのが肝。再ビルドのたびにやり直すので、多いと必ず崩れる。
- クラウド変数は数値のみなので、文字列 payload は数字エンコードが要る
  （soul-beat は 65 桁固定の数字列。仕様は `score_board_site/docs/cloud-format.md`）。

*(2026-06-24 / 2026-07-13 / soul-beat)*

### 6.2 作品はネスト git リポジトリ、ただしツールへの相対 import に依存する

作品は `workspace/<name>/` で個別に `git init` して独立管理する（ツール側の `.gitignore` が
`/workspace/` を無視しているので衝突しない）。生成物（`output/*.sb3`）も追跡する方針。

**注意**: 作品の `project.ts` は `../../src/...` でツール本体に相対 import しているので、
**作品リポジトリ単独では `npm run sb3` も `npm run preview` も動かない**。clone 後に
scratch-dsl ツリーの `workspace/<name>/` へ置く運用を作品 README に明記すること。

*(2026-06-23 / soul-beat)*

## 7. 進め方（プロセス）の教訓

### 7.1 サブエージェントへの委譲は「メイン文脈を守るが総消費は増やす」

サブエージェントは毎回コールドスタートでリポを再探索し、E2E 検証で transcript が肥大するため、
**1 タスクで 200k〜560k tokens** かかる。着手前にコストを見積もる:

- **1〜2 ファイルの明快な変更・低リスクはメインで直接編集する**（コールドスタート回避が安い）。
  大規模・探索が必要・実 VM E2E が本質的に要るものだけ委譲する。
- 検証は「デフォルト非回帰 + 新挙動の要点 1 つ」に絞る。網羅 E2E も大量スクショも要求しない
  （スクショの画像レビュー自体がトークンを食う）。
- 関連変更は 1 エージェントにまとめる。継続は新規 spawn ではなく文脈を保ったまま再開する。
- 実行中のエージェントが触るファイルはメイン側で編集しない（競合する）。

### 7.2 仕様が固まる前に着手しない

最大の無駄は「仕様未確定のまま実装 → 丸ごと再実装」だった（soul-beat の LANE LENGTH を
上部/下部で 2 回実装、2key の色と反応を後から訂正）。曖昧な点は**手を動かす前に潰す**。

### 7.3 重い実機検証はユーザーに委ねる分担を提案する

実機 E2E 検証のサブエージェントは 1 回 400k+ tokens かかることがある（遷移バグ 3 件の検証で
約 432k）。検証がトークン高になりそうなら、着手前に
**「メイン側で修正 + 精密な再現手順を用意し、実機テストはユーザーが行う」**分担を提案する。

### 7.4 理論より実測

このリポジトリで最も繰り返し効いた教訓。フェンシングのクランプ位置（§1.5）も、
グリフの字形崩れ（§5.5）も、**事前の計算・推論が実測と食い違った**。
Scratch の挙動は実 VM で確かめられるのだから、諦める前・結論づける前に一度動かす。
